## 4. Current issues / bugs

### Build / deploy blocker

- **`npm run build` fails** with Turbopack: **conflicting `route.ts` and `page.tsx` at the same segment**  
  **`/order-success/[id]`**:  
  `app/(store)/order-success/[id]/route.ts` **and** `app/(store)/order-success/[id]/page.tsx`.  
  Next.js 16 does not allow both for the same path. **Until one is merged into the other (e.g. single Route Handler that redirects after POST, or different path), production builds will not succeed.**

### PayU end-to-end

**Cannot assert “working in production”** — build fails, and the following code-level risks exist:

1. **Success path design**: PayU POSTs to **`surl`** → **`route.ts`** forwards `FormData` to **`/api/payments/payu/webhook`** → on OK redirects to **`/order-success/[id]`** (GET page). This is the right *idea* for PayU’s POST return.
2. **Reverse hash verification** (`lib/payu.ts` `verifyPayUHash`): PayU’s documented pattern is  
   `sha512(SALT|status||||||udf5|udf4|udf3|udf2|udf1|email|firstname|productinfo|amount|txnid|key)`  
   (six pipes between **`status`** and **`udf5`** — i.e. **six** empty fields).  
   The implementation uses **five** empty strings between `status` and `udf5`, which yields **one fewer** empty segment than the official formula. **Symptom if wrong: webhook returns HTTP 400 `"Invalid hash"`**, then **`order-success` `route.ts`** redirects to **`/payment-failed`** because `!webhookRes.ok`.
3. **`lib/payu.ts` `generatePayUHash`** logs **full hash strings** (including material derived from secret salt) to **server logs** — security and noise issue in production.
4. **`verifyPayUHash`** uses module-level **`PAYU_KEY`** / **`PAYU_SALT`** without the same **`.trim()`** used in request hash generation — trailing whitespace in env could cause **intermittent hash mismatches**.

### Console / production / UX

- **No access to production browser console or Vercel logs from this handoff** — cannot confirm live console errors.
- **Middleware deprecation warning** at build time: Next 16 reports **`middleware` file convention is deprecated** in favor of **`proxy`** (framework message only).
- **OTP route** (`app/api/otp/send/route.ts`) contains **`console.log`** calls including template id / phone flow — noisy and potentially sensitive in logs.

### Broken links / images / forms

- Not exhaustively crawled; **canonical / metadata URLs** point to **`https://www.nauvaraha.com`** in several pages while **`next-seo.config.ts`** uses **`https://nauvarah.com`** — **SEO / social duplication risk**, not necessarily a runtime 404 inside the app.
- **Customer complaints**: **not stored in repo** — unknown.

### Performance

- No Lighthouse or Vercel analytics data in-repo — **cannot** label specific slow pages from code alone.
